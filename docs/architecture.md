# LMC Infrastructure – Linux Server Administration Assignment

## Architecture Overview

```
Internet
    │
    ▼  :80
┌──────────┐
│  VM1     │  HAProxy  (load balancer / reverse proxy)
│ 10.0.0.10│
└────┬─────┘
     │ round-robin (health-checked)
  ┌──┴──────────────────┐
  ▼                     ▼
┌──────────┐       ┌──────────┐
│  VM2     │       │  VM3     │
│ web01    │       │ web02    │
│10.0.0.11 │       │10.0.0.12 │
│ Nginx    │       │ Nginx    │
│ PHP-FPM  │       │ PHP-FPM  │
└────┬─────┘       └────┬─────┘
     └──────┬───────────┘
            ▼ :3306
       ┌──────────┐
       │  VM4     │
       │  db      │
       │10.0.0.20 │
       │ MariaDB  │
       └──────────┘
```

## IP Addressing

| VM   | Hostname | Private IP | Role                    |
|------|----------|------------|-------------------------|
| VM1  | haproxy  | 10.0.0.10  | HAProxy reverse proxy   |
| VM2  | web01    | 10.0.0.11  | Nginx + PHP-FPM         |
| VM3  | web02    | 10.0.0.12  | Nginx + PHP-FPM         |
| VM4  | db       | 10.0.0.20  | MariaDB database server |

All instances share a single `/24` public subnet (`10.0.0.0/24`) inside a VPC (`10.0.0.0/16`).  
Each instance also receives a public IP for administrative SSH access.

## Security Model

```
Internet → HAProxy SG  : 80/tcp (HTTP), 8404/tcp (stats), 22/tcp (SSH)
HAProxy  → Web SG      : 80/tcp only
Web SG   → DB SG       : 3306/tcp only
DB SG    ← Internet    : blocked (only 22/tcp for admin SSH)
```

Additional hardening on every node:
- Direct root SSH login disabled (`PermitRootLogin no`)
- Admin user `sysadmin` with key-based auth only, passwordless sudo
- Encrypted EBS root volume (GP3, AES-256)
- IMDSv2 required (blocks SSRF attacks against instance metadata)
- `firewalld` with allowlist rules mirroring security-group rules (defence in depth)
- Time synchronised via `chronyd`

## Deployment

### Prerequisites

```bash
node --version   # ≥ 18
cdk  --version   # ≥ 2.97.0
aws  --version   # any recent
aws configure    # set account credentials + region
```

### Deploy

```bash
git clone <repo-url>
cd lmc
npm install

# One-time bootstrap (per account/region)
cdk bootstrap

# Preview what will be created
cdk diff

# Deploy both stacks
cdk deploy --all
```

CDK deploys two CloudFormation stacks:
- **LmcNetwork** – VPC, subnet, internet gateway, security groups
- **LmcServers** – 4 EC2 instances, key pair

### Retrieve the SSH Key

The private key is generated by CDK and stored in AWS SSM Parameter Store.  
The exact command is printed as a CloudFormation output (`SshKeyCommand`).

```bash
# Get it from the CDK output, or manually:
KEY_ID=$(aws ec2 describe-key-pairs --key-names lmc-keypair --query 'KeyPairs[0].KeyPairId' --output text)
aws ssm get-parameter \
    --name "/ec2/keypair/${KEY_ID}" \
    --with-decryption \
    --query Parameter.Value \
    --output text > lmc-key.pem
chmod 600 lmc-key.pem
```

### SSH Access

```bash
# HAProxy (public-facing)
ssh -i lmc-key.pem sysadmin@<HAPROXY_PUBLIC_IP>

# Web01 / Web02 / DB
ssh -i lmc-key.pem sysadmin@<WEB01_PUBLIC_IP>
ssh -i lmc-key.pem sysadmin@<WEB02_PUBLIC_IP>
ssh -i lmc-key.pem sysadmin@<DB_PUBLIC_IP>
```

Public IPs are printed as CloudFormation outputs after `cdk deploy`.

## Verification Steps

### 1 – Application through HAProxy

```bash
HAPROXY_IP=<from cdk output>

# Basic HTTP request
curl -s http://${HAPROXY_IP}/

# Health endpoint
curl -s http://${HAPROXY_IP}/health | jq .

# Load-balancing: run 4 times and observe alternating hostnames
for i in 1 2 3 4; do
  curl -s http://${HAPROXY_IP}/ | grep -oP '(?<=Hostname: )[\w-]+'
done
```

### 2 – Database is not accessible from the internet

```bash
# Should time out / be refused (DB port blocked by security group)
nc -zv <DB_PUBLIC_IP> 3306      # expected: connection refused / timeout

# Works from a web server (inside the security group)
ssh -i lmc-key.pem sysadmin@<WEB01_PUBLIC_IP> \
    "mysql -h 10.0.0.20 -u appuser -pAppPass123! appdb -e 'SELECT COUNT(*) FROM visits;'"
```

### 3 – Failover when one web server is stopped

```bash
# Stop Web01
aws ec2 stop-instances --instance-ids <WEB01_INSTANCE_ID>

# Keep hitting the load balancer – should still return 200 from web02
watch -n1 "curl -s http://${HAPROXY_IP}/ | grep Hostname"

# HAProxy stats page shows web01 as DOWN
curl -s http://${HAPROXY_IP}:8404/stats  # or open in browser
```

### 4 – Logs for troubleshooting

```bash
# HAProxy access log
ssh -i lmc-key.pem sysadmin@<HAPROXY_IP> "sudo journalctl -u haproxy -f"

# Nginx access / error log
ssh -i lmc-key.pem sysadmin@<WEB01_IP> "sudo tail -f /var/log/nginx/app_access.log"
ssh -i lmc-key.pem sysadmin@<WEB01_IP> "sudo tail -f /var/log/nginx/app_error.log"

# PHP-FPM log
ssh -i lmc-key.pem sysadmin@<WEB01_IP> "sudo journalctl -u php-fpm -f"

# MariaDB log
ssh -i lmc-key.pem sysadmin@<DB_IP>    "sudo journalctl -u mariadb -f"

## Database Details

| Item           | Value        |
|----------------|--------------|
| Database name  | appdb        |
| App user       | appuser      |
| App password   | AppPass123!  |
| Root password  | RootPass123! |

Tables:
- `users(id, username, email, created_at)` – pre-seeded with 2 rows
- `visits(id, server_hostname, visited_at)` – written on every page load

## HAProxy Details

| Setting             | Value                   |
|---------------------|-------------------------|
| Balance algorithm   | round-robin             |
| Health check URL    | `GET /health.php`       |
| Check interval      | 5 s                     |
| Mark DOWN after     | 2 consecutive failures  |
| Mark UP after       | 3 consecutive successes |
| Stats page          | `:8404/stats`           |
| Stats credentials   | admin / admin123        |

The `/health` path is routed to a dedicated HAProxy backend so monitoring tools can check
health without being load-balanced the same way as normal traffic.

## Destroy the Stack

```bash
cdk destroy --all
```

CloudFormation will terminate all EC2 instances and delete the VPC.  
The EBS volumes are deleted automatically (`deleteOnTermination: true`).

## Known Limitations and Possible Improvements

| Limitation | Improvement |
|---|---|
| All instances in a single AZ (single-AZ) | Spread web/HAProxy across 2+ AZs for HA |
| Web/DB in public subnet with public IPs | Move web/DB to private subnets + NAT gateway |
| DB credentials hardcoded in userdata | Use AWS Secrets Manager |
| HAProxy is a single point of failure | Add a standby HAProxy with Keepalived/VIP |
| HTTP only | Add TLS with ACM + ALB, or Let's Encrypt on HAProxy |
| t3.micro instances | Right-size based on load testing |
| Single MariaDB node | Add replica or move to RDS Multi-AZ |
| firewalld rules duplicate security group rules | Defence in depth is good; could be automated with Ansible |
| Sticky sessions not configured | Add `cookie` directive in HAProxy backend for stateful apps |
